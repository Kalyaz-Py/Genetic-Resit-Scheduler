# Система автоматизированного формирования расписания пересдач студентов

**Профессиональное десктоп-приложение (WPF / .NET / MS SQL Server), решающее задачу эффективного распределения ресурсов учебного заведения при организации академических пересдач.**

---

## 🚀 Ключевая ценность и фишка проекта
Главная проблема составления расписания пересдач — огромная вариативность и пересечение ограничений. Это классическая **NP-трудная задача оптимизации**. Человеку (диспетчеру) требуются часы, чтобы составить бесконфликтный план.

В основе ядра данной системы лежит кастомная реализация **эволюционного (генетического) алгоритма**, который находит оптимальное или близкое к оптимальному решение за несколько секунд.

### Как работает алгоритм в проекте:
1. **Хромосома (Особи):** Каждая особь представляет собой один из вариантов полного расписания пересдач.
2. **Фитнес-функция (Функция приспособленности):** Оценивает качество расписания. За каждое нарушение ограничений (жестких или мягких) особь получает штрафные баллы. Цель алгоритма — минимизировать штраф.
3. **Эволюционный цикл:** Применяются операторы **селекции** (отбор лучших вариантов), **скрещивания (кроссинговера)** для обмена успешными блоками расписания и **мутации** (случайные изменения слотов/кабинетов для выхода из локальных экстремумов).

### Учитываемые ограничения (Constraints):
*   **Конфликты преподавателей:** Один преподаватель физически не может принимать две разные пересдачи в один и тот же временной слот.
*   **Конфликты аудиторного фонда:** Исключено назначение разных экзаменов в один кабинет в одно время.
*   **Пересечение с основным расписанием:** Алгоритм анализирует текущую занятость аудиторий и преподавателей в сетке регулярных пар колледжа.
*   **Специфика дисциплин:** Учитывается соответствие кабинета принимаемой дисциплине (например, компьютерный класс для программирования).
*   **Временные лимиты:** Ограничение доступных слотов (например, только после основных пар).

## 🛠 Стек технологий и архитектура

Приложение спроектировано с учетом паттернов Clean Code и четким разделением ответственности (Separation of Concerns):
*   **Язык программирования:** C#
*   **Платформа:** .NET Framework
*   **Паттерн архитектуры UI:** MVVM (Model-View-ViewModel) — обеспечивает полную независимость вычислительного ядра алгоритма от графического интерфейса WPF.
*   **База данных:** MS SQL Server (Реляционная структура, оптимизированные связи).
*   **Слой доступа к данным (DAL):** ADO.NET + LINQ (Использование параметризованных запросов и транзакций для безопасности и целостности данных).

## 📊 Архитектура базы данных

В проекте используется реляционная база данных MS SQL Server. Схема полностью нормализована (3НФ), что исключает дублирование данных и обеспечивает целостность при работе алгоритма генерации.

### Ключевые таблицы и сущности:
*   **`Debts` (Задолженности):** Центральная таблица связующей логики. Хранит информацию о долгах студентов (`StudentID`), дисциплинах (`DisciplineID`), назначенных преподавателях (`TeacherID`), статусе отработки (`DebtStatus`, `IsCleared`) и финальной оценке.
*   **`ResitSchedules` (Расписание пересдач):** Результат работы генетического алгоритма. Содержит ссылки на конкретную задолженность (`DebtID`), назначенную дату (`ScheduledDate`), аудиторию (`ClassroomID`) и номер попытки.
*   **`Schedules` и `TimeSlots`:** Сетка регулярных занятий колледжа. Алгоритм использует её для валидации жестких ограничений (Hard Constraints), проверяя занятость аудиторий и преподавателей на обычных парах, чтобы избежать накладок.
*   **`Teachers` и `TeacherDisciplines`:** Данные о преподавательском составе и таблица-преемник для реализации связи "многие-ко-многим", определяющая, какие предметы имеет право принимать конкретный преподаватель.
*   **`Students` и `Groups`:** Учет студентов, их номеров зачетных книжек (`StudentCardNumber`), курсов и отделений для формирования аналитики.
*   **`Classrooms`:** Аудиторный фонд с указанием вместимости (`Capacity`) и типа кабинета (`ClassroomType`), что позволяет алгоритму автоматически подбирать специализированные классы (например, компьютерные).
*   **`Users` и `Roles`:** Ролевая модель (`RoleID`, `Name`) для разграничения прав доступа к интерфейсу генерации (Администратор / Диспетчер / Преподаватель).
